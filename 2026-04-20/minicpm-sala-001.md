## NVFP4 KV Cache Feasibility Analysis

### What I found after deep-diving the codebase

**The infrastructure exists, but there are critical gaps for our config:**

| Component | FP4 Support? | Detail |
|-----------|-------------|--------|
| Server arg `--kv-cache-dtype fp4_e2m1` | Yes | Sets `torch.float4_e2m1fn_x2` (requires **PyTorch 2.8+**) |
| Memory pool `MHATokenToKVPoolFP4` | Yes | Stores K/V as packed uint8 (half size) + scale buffers, dequantizes on read |
| FlashInfer backend | **NO** | `should_use_tensor_core()` doesn't handle FP4; `data_type=FP4` passed to `begin_forward()` but `get_kv_buffer()` returns dequantized bf16 â€” **dtype mismatch** |
| TRT-LLM MLA backend | Yes | Native FP4 support with uint8 buffers |
| MiniCPM sparse backend | **NO** | Only bridges fp8â†’bf16, not fp4 |
| Per-layer mixed-precision KV | **NO** | Single global `kv_cache_dtype` for all layers |

### Key problems

1. **FlashInfer mismatch**: `begin_forward()` is configured with `data_type=float4_e2m1fn_x2`, but `get_kv_buffer()` returns **dequantized bf16** tensors. FlashInfer's CUDA kernels would interpret the data incorrectly or crash.

2. **Even if it works, pure FP4 = accuracy death**: Champion got 80% â†’ **75%** with pure FP4 KV. For us: 80% â†’ ~75% = normalized ~94% â†’ **C = 0 (eliminated)**. Pure FP4 is unviable without mixed-precision per-layer support.

3. **Mixed-precision per-layer KV cache requires significant new code**: New memory pool class, per-layer dtype tracking, modified attention backends, sensitivity analysis tooling. This is the champion's secret sauce but it's a multi-day implementation effort.

4. **PyTorch version uncertainty**: If fcloud has PyTorch < 2.8, `torch.float4_e2m1fn_x2` doesn't exist and it silently falls back to model dtype (bf16). The FP4 path is never activated.

### Recommendation: Skip FP4 KV Cache, Focus on EAGLE3

| Option | Effort | Expected Gain | Risk |
|--------|--------|--------------|------|
| **A: Quick FP4 KV test** | 5 min | Likely crashes or C=0 | Wastes fcloud time |
| **B: Mixed-precision KV** | 3-5 days | ~5-10% speed if it works | High complexity, uncertain accuracy |
| **C: EAGLE3 speculative decode** | 1-2 days | **2-3x decode speedup** | Draft model already trained, medium risk |

**EAGLE3 is the clear winner:**
- Draft model already exists at `/root/models/eagle3_draft_minicpm/` (612MB)
- No accuracy loss (speculative decoding is lossless)
- Directly improves S1 (40% weight), which is our biggest bottleneck
- Code is on the `eagle3-spec-decode` branch, ready to integrate
