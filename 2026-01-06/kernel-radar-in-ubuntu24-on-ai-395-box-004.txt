What you’re seeing is the key clue:


run_command: .../vscode .../extensions/git/dist/askpass.sh 'Password for ...'
So git send-email is not reading the password from your terminal — it’s delegating to VS Code’s askpass.sh helper. When that helper doesn’t complete cleanly, it looks like a “hang”.

Fix: force a real terminal password prompt (bypass VS Code askpass)
Abort the stuck run: press Ctrl+C.

Re-run the send command with askpass disabled:


cd ~/mylinux/linuxenv -u GIT_ASKPASS -u SSH_ASKPASS GIT_TERMINAL_PROMPT=1 \git send-email outgoing/0001-scripts-atomic-fix-kerneldoc-spelling-in-try_cmpxchg.patch \  --to will@kernel.org \  --cc peterz@infradead.org \  --cc boqun.feng@gmail.com \  --cc mark.rutland@arm.com \  --cc gary@garyguo.net \  --cc linux-kernel@vger.kernel.org \  --confirm=always
Now it should prompt in the terminal for password; paste the 16‑char Gmail App Password.

If you want more visibility while you test, add --smtp-debug=1 to that command.

If it still “hangs”
Run this to see whether your environment is still injecting askpass:


env | grep -E 'GIT_ASKPASS|SSH_ASKPASS|GIT_TERMINAL_PROMPT' || true
Optional (but recommended): don’t CC yourself automatically
Right now git is auto-adding you as CC due to From: and Signed-off-by:. That’s harmless, but if you prefer not to, set:


git config --global sendemail.suppresscc all
Then resend.
