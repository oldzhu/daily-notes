Packaging-wise you don’t get to run arbitrary shell scripts during a pip install (wheels are just zipped metadata + bytecode), so the usual way to “prompt” users is via the console entry points or via package import-time warnings.
Two practical options:
Post-install print in entry point – when the user runs dbgcopilot/dbgagent, detect missing tools (shutil.which) and print a short warning plus a URL (https://github.com/oldzhu/dbgcopilot/docs/install.md#prerequisites or similar) explaining how to install them. That message can mention the exact shell command we already display inside /use.
Documentation badge/link – add a badge or note inside install.md and README.md that calls out “Missing Delve/JDB/Radare2? See docs…” and use pip show dbgagent/pip show dbgcopilot to surface the URL. You could even expose a dbgtool helper (console script) that runs the checks and prints the doc link.
If you need something during pip install, the only hook is within setup.py/pyproject via setup_requires or build scripts (which run on build time, not install time) – but those can’t prompt end users on install. So the entry-point runtime warnings plus doc links are the safest approach for now.
