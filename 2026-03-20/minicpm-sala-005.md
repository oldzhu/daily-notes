这篇冠军分享和我们现在遇到的问题，相关性很高。它基本验证了我们最近的判断方向里，有两点是对的，一点需要修正。

**关键信息**

1. 他把重点从“继续榨速度”转到了“精度抖动治理”和“校准数据集优化”。
这和你现在的处境一致。你现在的主要问题已经不是单纯速度，而是量化后正确率不稳、提交结果不可预判。

2. 他明确说公开正确性集一共 150 条，5 种任务各 30 条。
这和我们当前看到的数据一致，也说明“只按前 N 条顺序截取”很可能是有偏的。

3. 最重要的一点：
他不建议让校准集里“语义相关性低的数据”占比过高，尤其提到随机单词、UUID 这类内容可能对校准有负面影响，应该适当降低比例，用更多有连续语义的数据替代。

这点对我们当前很关键。

**这对我们当前工作的直接含义**

我认为它说明：我们刚做的默认 `stratified`，方向对了一半，但还不够好。

为什么说只对了一半：

1. 对的部分：
我们已经从“顺序前 N 条”走出来了，开始按 task 和长度分层，这比盲选前 32 条合理。

2. 不够好的部分：
我们现在的 `stratified` 只看了 task 和长度，没有看“语义类型”。

而冠军分享指出，校准集不应该机械复刻公开评测集分布，尤其不要让低语义相关样本占比太高。

这正好解释你现在的结果为什么还是差：
- 32 条默认 stratified 里，长上下文类占比很高
- 其中一部分可能恰好是低语义检索/关键词抽取类样本
- 这些样本对 preserving real reasoning quality 未必有帮助，甚至可能有害

**和我们当前实验现象怎么对应**

你刚跑的默认 `stratified + 32` 只有 `76.00%`，说明至少有一种情况成立：

1. 你的 last-good quant model 的确更接近 `sequential + 8/16`
或者
2. 当前 `stratified` 把过多预算给了“对 GPTQ 校准帮助不大的样本类型”
或者
3. 两者都成立

而冠军分享更支持第 2 点。

**我现在对下一步的判断**

我不建议你继续沿着“通用 stratified 默认策略”盲试更多轮。

更合理的路线应该分成两步。

**第一步：先回放 last-good 行为**
这一步是为了找回基线，不是为了最终方案。

你刚才说你记得 last-good 可能是 `8` 或 `16` 个 sequential 样本量化出来的。
我认为这件事优先级很高，因为它能回答一个关键问题：

- 现在的精度问题，到底是“最近改坏了”
- 还是“其实以前那个看起来好的结果本身就有偶然性”

所以这一步我赞成你去确认：
- `SOAR_GPTQ_CALIBRATION_SAMPLING=sequential`
- `SOAR_GPTQ_CALIBRATION_SAMPLES=8`
- `SOAR_GPTQ_CALIBRATION_SAMPLES=16`

如果其中一个能把正确率拉回去，我们就知道：
- 当前问题主要在 calibration set composition
- 而不是 GPTQ 整体链路已经不可救

**第二步：不要用“全量分布复制”，而要做“语义过滤后的校准采样”**
这是这篇分享给我们的最大启发。

我现在更倾向于把 calibration 样本分成两大类：

1. 高语义连续性样本
例如：
- mcq
- 有明确推理链或知识关联的 qa
- 有连续自然语言上下文的长文本问答

2. 低语义相关样本
例如：
- UUID / magic number / 随机键值检索
- 过度模板化、主要考记忆定位而不是语义建模的样本

冠军的观点是：
- 第二类不应该在校准集中占太高比例

这和我们现在的默认 `stratified` 最大区别是：
- 现在我们把这些都当成普通 task bucket
- 但实际上它们对量化校准价值不一样

**所以我建议后续不要直接“按 task+length 自动分层”作为最终方向**
更好的方向应该是：

1. 先确认 last-good 的 sequential 8/16 是否能回得去
2. 如果能回去，再做第二个 feature：
   做“语义过滤/降权”的 calibration sampling

也就是：
- 保留少量 UUID / magic-number / keyword-extraction 样本
- 但显著增加连续语义样本的比例

**这篇文章还给我们另一个重要启发**

他搭了一个基于 logprob 的快速精度评测工具，用 BF16 baseline 和量化模型做输出分布对比。

这个点我认为非常值得我们参考，原因是：

1. 你现在每次跑完整 eval_model.py 成本太高
2. 结果还会受到生成长度和 decode 抖动影响
3. 他的方案更适合做“提交前筛掉明显异常配置”的本地快检

但要注意，他自己也说了：
- 这个工具主要感知 prefill 阶段误差
- 不覆盖 decode 阶段误差传播

所以它不是替代官方正确率评测，而是：
- 提前筛除坏配置
- 降低“每天 3 次提交靠运气”的风险

**我对我们接下来怎么做的建议**

按优先级排序，我建议：

1. 先做你现在正在做的事：
   复测 `sequential + 8/16`
   这是最高优先级。

2. 如果 `sequential + 8/16` 真的更好：
   下一步不要继续优化当前默认 `stratified`
   而是单独做一个新 feature：
   “语义过滤的校准采样”
   核心思想来自这篇分享：
   降低低语义样本比例，增加连续语义样本比例。

3. 再下一步：
   搭一个轻量的本地快速精度筛查工具
   不一定一开始就做完整 logprob KL/cosine 体系，但可以先做最小版本。

**我现在最重要的结论**

这篇冠军分享让我更确信两件事：

1. 我们当前的主战场确实应该是 accuracy stability，不是继续盲目压速度。
2. 当前默认 `stratified` 不是终点，因为它只做了“分布均衡”，还没有做“语义价值筛选”。

所以，如果你问我“看完这篇之后，最该怎么推进”：

- 短期：
  先验证 `sequential + 8/16` 是否能复现 last-good
- 中期：
  把 calibration sampling 从“task/length stratified”升级成“语义过滤 + 轻量分层”
- 后期：
  再补一个 logprob-based 快速精度筛查工具，减少完整 eval 的试错成本
